## Objective

The 2026-07-14 fleet finding (bench_serving --dataset-name random with the upstream default --random-range-ratio 0.0 draws prompt AND output lengths uniform in [1, N], so every labeled-depth row measured ~half depth — or, at num-prompts 1, ANY depth in [1, N]) never propagated to the M4 repo: all four bench_serving entry points are unpinned. Pin the ratio, record server-verified actual input tokens per point, and mark every existing bench_serving depth-labeled artifact LEGACY-DEPTH-SUSPECT so the rig has a truthful instrument — a prerequisite for the long-context item (M4-A). Scope is the bench_serving instrument only; the repo's separate /v1/completions long_context instrument is already server-verified and out of scope.

## Constraints

- oom_guard.sh MANDATORY for any >=32K point (macOS has no OOM killer; an OOM stalls the box). MEM_FRAC is a fraction of TOTAL system RAM: never above 0.7, use 0.4 for 32K+ (CLAUDE.md).
- One mechanism at a time: land the pin as its own commit before any re-measure; do not fold it into bisect or rebase work. Re-measures unavoidably move two variables (pin + sglang version): record the sglang commit in every new row and do not present re-measured rows as a depth-only correction of legacy rows.
- Do not conflate the two depth instruments: the range-ratio pin and its LEGACY note apply only to bench_serving/context_sweep artifacts; the /v1/completions long_context instrument (qwen36 results.json, long_context_*.txt) is server-verified, immune, and must not be pinned, edited, or re-labeled with the range-ratio mechanism.
- Depth/decode numbers only from server-verified actual token counts (bench_serving 'Total input tokens' / usage.prompt_tokens / server-log ground truth) — never client TPOT, never unpinned random rows.
- Detach any >30min bench via the setsid pattern; single GPU workload at a time on the M4 box.
- Negative results (OOM-below-16K, smoke shortfall) are findings — record with receipts in benchmarks/, do not discard.


## Risks

- Post-pin bench_serving numbers shift systematically 'worse' (real tokens ~2x legacy average): anyone diffing new vs old rows without reading the LEGACY marker will see a phantom regression — the instrument-split marker and baseline invalidation are the mitigation, land them in the same commit as the pin.
- Legacy 256K-labeled context_sweep rows (e.g. coder-next-80b) were single uniform draws at num_prompts=1 — per-row correction magnitude is unknowable; they can only be flagged, and deep rows cannot be re-measured until the 32K ceiling is fixed. (This unknowability does NOT apply to qwen36: its /v1/completions instrument recorded the real prompt_tokens per point.)
- Executor hazard: the M4 repo's local scripts/bench/bench_long_context.py shares the donor's name/path but is a different, working, server-verified instrument. Applying the donor line-edits to it would corrupt it — step 4's guard clause and the deliverables 'NOT edited' note are the mitigation.
- If the historical '128K fit' claim classifies as bench_serving-random-derived, the true historical depth may have been far less than 128K, shrinking the regression the long-context item is chartered to explain — feed the verdict there rather than editing its charter unilaterally.
- Version confound: re-measures run on the current pin while legacy rows are from earlier pins; without recording the commit, a reader could misread a version-driven delta as a depth-pin delta. Mitigation: sglang commit stamped in every new results.json, explicit note in LEGACY-DEPTH-SUSPECT.md.
- Argparse prefix matching: skipping step 3 leaves --random-input one upstream option-addition away from an ambiguity crash at the next rebase.
- Pinned re-measure doubles per-point token load vs legacy runs: expect longer prefills and possibly earlier OOM points; budget wall-clock accordingly and keep the guard threshold defaults.
